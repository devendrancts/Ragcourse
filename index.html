Core RAG Foundation

Retrieval-Augmented Generation (RAG) enhances LLMs by providing relevant external knowledge before generating a response. Instead of relying solely on training data, RAG grounds the model's response in actual retrieved documents, dramatically reducing hallucinations.

💡
Critical Prompt Design: The prompt template MUST instruct the LLM: "Answer based only on the following context. If the context does not contain the answer, say I don't know." This is the primary defense against hallucination.

Complete RAG Chain Setup

python
from langchain_openai import ChatOpenAI, OpenAIEmbeddings
from langchain_chroma import Chroma
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_core.runnables import RunnablePassthrough

# Create embeddings and vector store
embeddings = OpenAIEmbeddings(model="text-embedding-3-small")
vectorstore = Chroma.from_documents(chunks, embeddings)
retriever = vectorstore.as_retriever(search_kwargs={"k": 3})

# Prompt template
template = """Answer based ONLY on this context. If unsure, say "I don't know."
Context: {context}
Question: {question}"""
prompt = ChatPromptTemplate.from_template(template)

# Build the chain
rag_chain = (
    {"context": retriever | format_docs, "question": RunnablePassthrough()}
    | prompt
    | ChatOpenAI(model="gpt-4", temperature=0)
    | StrOutputParser()
)

response = rag_chain.invoke("What is the refund policy?")

Adding Citations

Citations let users verify the sources behind each answer. This function formats retrieved documents with their source metadata for transparency.

python
def format_docs_with_sources(docs):
    formatted = []
    for i, doc in enumerate(docs, 1):
        source = doc.metadata.get("source", "Unknown")
        page = doc.metadata.get("page", "N/A")
        formatted.append(f"[Source {i}: {source}, Page {page}]\n{doc.page_content}")
    return "\n\n---\n\n".join(formatted)

Failure Mode 1: Bad Chunking

Solved by applying the 4 chunking variables from Section 2. Choose the right chunk size, overlap, split strategy, and content-type-aware splitter.

Failure Mode 2: Embedding Mismatch

🔒 Rule 1: Same Model

Always use the exact same embedding model and version for both indexing and querying. Mixing models produces meaningless similarity scores.

📏 Rule 2: Same Dimensions

If using dimension reduction, use the same dimension at index and query time. A 512-dim query vector cannot match a 1536-dim index vector.

📌 Rule 3: Version Lock

Pin your embedding model version. When the provider updates the model, re-index everything or keep using the old version.

⚠️
Critical Insight: 90% of RAG failures are retrieval failures, not generation failures. ALWAYS test your retrieval separately before blaming the LLM.

Failure Mode 3: Retrieval Noise — Hybrid Search

Pure vector search struggles with exact keywords, product codes (SQ7742X), acronyms (WCAG), and error codes. Hybrid search combines semantic understanding with keyword matching.

python
from langchain_community.retrievers import BM25Retriever
from langchain.retrievers import EnsembleRetriever

vector_retriever = vectorstore.as_retriever(search_kwargs={"k": 3})
bm25_retriever = BM25Retriever.from_documents(chunks, k=3)

hybrid = EnsembleRetriever(
    retrievers=[vector_retriever, bm25_retriever],
    weights=[0.7, 0.3]  # 70% semantic, 30% keyword
)
results = hybrid.invoke("What is error SQ7742X?")
⚠️
BM25 Limitation: BM25 does NOT support incremental updates. It must be rebuilt every time new documents are added. Adds only 20-50ms latency but massively improves accuracy.

Failure Mode 4: Context Overflow — Token Budgeting

A single request with a 200-page contract can blow your entire daily token budget. Estimate and check before the API call.

python
class TokenBudget:
    def __init__(self, max_per_request=4000, max_per_day=1_000_000):
        self.max_per_request = max_per_request
        self.max_per_day = max_per_day
        self.used_today = 0

    def estimate_tokens(self, text: str) -> int:
        return int(len(text) * 1.3)  # ~1.3 tokens per char for English

    def check(self, text: str) -> bool:
        est = self.estimate_tokens(text)
        if est > self.max_per_request:
            raise ValueError(f"Request too large: {est} tokens")
        if self.used_today + est > self.max_per_day:
            raise ValueError("Daily budget exceeded")
        return True  # OK to proceed

    def record(self, input_tokens: int, output_tokens: int):
        self.used_today += input_tokens + output_tokens

budget = TokenBudget()
budget.check(context_text)  # Check BEFORE the API call = no cost

Cost Optimization Strategies

📈 Reduce Dimensions

1536 ➔ 512 dimensions = 30-66% storage savings with minimal quality loss. OpenAIEmbeddings(dimensions=512)

📥 Quantization

float32 ➔ int8 = 50-75% storage savings. Most vector databases support this natively.

💾 Response Caching

Cache common queries = 50-80% API cost savings. Use SHA-256 hash of normalized query as cache key with TTL.

📦 Batch Embedding

Embed in batches, not one-by-one = 20-40% overhead savings. Use embed_documents() not embed_query() in loop.

Security Pipeline

python
import re

class InputSanitizer:
    INJECTION_PATTERNS = [
        re.compile(r"ignore\s+(all\s+)?previous\s+instructions", re.I),
        re.compile(r"you\s+are\s+now", re.I),
        re.compile(r"system\s*prompt\s*:", re.I),
        re.compile(r"forget\s+(everything|all)", re.I),
    ]

    def check(self, text: str) -> tuple[bool, str]:
        for p in self.INJECTION_PATTERNS:
            if p.search(text):
                return False, "Blocked: injection detected"
        return True, "OK"

class PIIDetector:
    PATTERNS = {
        "email": re.compile(r'\b[\w.+-]+@[\w-]+\.[\w.]+\b'),
        "ssn":   re.compile(r'\b\d{3}-\d{2}-\d{4}\b'),
        "phone": re.compile(r'\b\(?\d{3}\)?[-.\s]?\d{3}[-.\s]?\d{4}\b'),
    }

    def mask(self, text: str) -> str:
        for pii_type, pattern in self.PATTERNS.items():
            text = pattern.sub(f"[{pii_type.upper()} REDACTED]", text)
        return text

def add_context_to_chunk(full_doc: str, chunk: str, doc_title: str) -> str:
    """Prepend LLM-generated context to a chunk before embedding."""
    prompt = f"""Add a 50-word context to this chunk from "{doc_title}".
Start with "This chunk is from..." and clarify pronouns/references.

Document excerpt: {full_doc[:3000]}
Chunk: {chunk}
Context:"""
    context = llm.invoke(prompt).content
    return f"{context}\n\n{chunk}"

# Apply at INDEX TIME (one-time cost: ~1-5 cents per doc)
contextualized = [add_context_to_chunk(doc, c, title) for c in chunks]

Graph RAG (Microsoft)

Traditional RAG retrieves isolated chunks. It cannot answer multi-hop questions like "Who are the competitors of companies that John advises?". Graph RAG extracts entities and relationships into a knowledge graph.

Knowledge Graph Example

John Smith works at Advisory LLC
Jane Doe founded Tech Corp
Bob Lee founded Data Inc
Jane Doe board member Global Fund

Multi-hop query: John ➔ advises Tech Corp ➔ competes with Data Inc ➔ Answer: Data Inc
