Core RAG Foundation

Retrieval-Augmented Generation (RAG) enhances LLMs by providing relevant external knowledge before generating a response. Instead of relying solely on training data, RAG grounds the model's response in actual retrieved documents, dramatically reducing hallucinations.

💡
Critical Prompt Design: The prompt template MUST instruct the LLM: "Answer based only on the following context. If the context does not contain the answer, say I don't know." This is the primary defense against hallucination.

Complete RAG Chain Setup

python
from langchain_openai import ChatOpenAI, OpenAIEmbeddings
from langchain_chroma import Chroma
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_core.runnables import RunnablePassthrough

# Create embeddings and vector store
embeddings = OpenAIEmbeddings(model="text-embedding-3-small")
vectorstore = Chroma.from_documents(chunks, embeddings)
retriever = vectorstore.as_retriever(search_kwargs={"k": 3})

# Prompt template
template = """Answer based ONLY on this context. If unsure, say "I don't know."
Context: {context}
Question: {question}"""
prompt = ChatPromptTemplate.from_template(template)

# Build the chain
rag_chain = (
    {"context": retriever | format_docs, "question": RunnablePassthrough()}
    | prompt
    | ChatOpenAI(model="gpt-4", temperature=0)
    | StrOutputParser()
)

response = rag_chain.invoke("What is the refund policy?")

Adding Citations

Citations let users verify the sources behind each answer. This function formats retrieved documents with their source metadata for transparency.

python
def format_docs_with_sources(docs):
    formatted = []
    for i, doc in enumerate(docs, 1):
        source = doc.metadata.get("source", "Unknown")
        page = doc.metadata.get("page", "N/A")
        formatted.append(f"[Source {i}: {source}, Page {page}]\n{doc.page_content}")
    return "\n\n---\n\n".join(formatted)

Failure Mode 1: Bad Chunking

Solved by applying the 4 chunking variables from Section 2. Choose the right chunk size, overlap, split strategy, and content-type-aware splitter.

Failure Mode 2: Embedding Mismatch

🔒 Rule 1: Same Model

Always use the exact same embedding model and version for both indexing and querying. Mixing models produces meaningless similarity scores.

📏 Rule 2: Same Dimensions

If using dimension reduction, use the same dimension at index and query time. A 512-dim query vector cannot match a 1536-dim index vector.

📌 Rule 3: Version Lock

Pin your embedding model version. When the provider updates the model, re-index everything or keep using the old version.

⚠️
Critical Insight: 90% of RAG failures are retrieval failures, not generation failures. ALWAYS test your retrieval separately before blaming the LLM.

Failure Mode 4: Context Overflow — Token Budgeting

A single request with a 200-page contract can blow your entire daily token budget. Estimate and check before the API call.

python
class TokenBudget:
    def __init__(self, max_per_request=4000, max_per_day=1_000_000):
        self.max_per_request = max_per_request
        self.max_per_day = max_per_day
        self.used_today = 0

    def estimate_tokens(self, text: str) -> int:
        return int(len(text) * 1.3)  # ~1.3 tokens per char for English

    def check(self, text: str) -> bool:
        est = self.estimate_tokens(text)
        if est > self.max_per_request:
            raise ValueError(f"Request too large: {est} tokens")
        if self.used_today + est > self.max_per_day:
            raise ValueError("Daily budget exceeded")
        return True  # OK to proceed

    def record(self, input_tokens: int, output_tokens: int):
        self.used_today += input_tokens + output_tokens

budget = TokenBudget()
budget.check(context_text)  # Check BEFORE the API call = no cost

Cost Optimization Strategies

📈 Reduce Dimensions

1536 ➔ 512 dimensions = 30-66% storage savings with minimal quality loss. OpenAIEmbeddings(dimensions=512)

📥 Quantization

float32 ➔ int8 = 50-75% storage savings. Most vector databases support this natively.

💾 Response Caching

Cache common queries = 50-80% API cost savings. Use SHA-256 hash of normalized query as cache key with TTL.

📦 Batch Embedding

Embed in batches, not one-by-one = 20-40% overhead savings. Use embed_documents() not embed_query() in loop.

Security Pipeline

python
import re

class InputSanitizer:
    INJECTION_PATTERNS = [
        re.compile(r"ignore\s+(all\s+)?previous\s+instructions", re.I),
        re.compile(r"you\s+are\s+now", re.I),
        re.compile(r"system\s*prompt\s*:", re.I),
        re.compile(r"forget\s+(everything|all)", re.I),
    ]

    def check(self, text: str) -> tuple[bool, str]:
        for p in self.INJECTION_PATTERNS:
            if p.search(text):
                return False, "Blocked: injection detected"
        return True, "OK"

class PIIDetector:
    PATTERNS = {
        "email": re.compile(r'\b[\w.+-]+@[\w-]+\.[\w.]+\b'),
        "ssn":   re.compile(r'\b\d{3}-\d{2}-\d{4}\b'),
        "phone": re.compile(r'\b\(?\d{3}\)?[-.\s]?\d{3}[-.\s]?\d{4}\b'),
    }

    def mask(self, text: str) -> str:
        for pii_type, pattern in self.PATTERNS.items():
            text = pattern.sub(f"[{pii_type.upper()} REDACTED]", text)
        return text

Re-ranking and Cross-Encoders

Initial retrieval with bi-encoders is fast but approximate. Re-ranking adds a second, more precise scoring stage using cross-encoders, boosting accuracy by 15-25%. Combined with contextual retrieval, it reduces failures by 67%.

Bi-Encoder vs. Cross-Encoder

▼

Why not just use cross-encoders? Running a cross-encoder on 500k docs would take ~83 minutes. That's why we use bi-encoders for fast initial retrieval, then cross-encoders only on the top candidates.

# Option 3: Custom cross-encoder with sentence-transformers
from sentence_transformers import CrossEncoder

model = CrossEncoder("cross-encoder/ms-marco-MiniLM-L-6-v2")

def rerank(query: str, docs, top_n: int = 3):
    pairs = [(query, doc.page_content) for doc in docs]
    scores = model.predict(pairs)
    ranked = sorted(zip(scores, docs), key=lambda x: x[0], reverse=True)
    return [doc for _, doc in ranked[:top_n]]

# Retrieve top 20, re-rank to top 3
candidates = vectorstore.as_retriever(search_kwargs={"k": 20}).invoke(query)
top_results = rerank(query, candidates, top_n=3)

Text Cleaning Pipeline

python
import re
import unicodedata

def clean_text(text: str) -> str:
    """Production text cleaning pipeline."""
    # 1. Unicode normalization
    text = unicodedata.normalize("NFC", text)

    # 2. Remove page numbers and headers
    text = re.sub(r'\n\s*Page \d+\s*\n', '\n', text)
    text = re.sub(r'\n\s*-\s*\d+\s*-\s*\n', '\n', text)

    # 3. Fix broken sentences (from PDF line wrapping)
    text = re.sub(r'(\w)-\n(\w)', r'\1\2', text)  # Re-join hyphenated words

    # 4. Collapse whitespace
    text = re.sub(r'\n{3,}', '\n\n', text)  # Max 2 newlines
    text = re.sub(r' {2,}', ' ', text)       # Max 1 space

    return text.strip()

# Apply to all loaded documents
for doc in documents:
    doc.page_content = clean_text(doc.page_content)

import hashlib
from pathlib import Path

def enrich_metadata(doc, file_path: str):
    """Add rich metadata to a document for filtering and citations."""
    path = Path(file_path)
    doc.metadata.update({
        "source": path.name,
        "file_type": path.suffix.lstrip("."),
        "content_hash": hashlib.md5(doc.page_content.encode()).hexdigest(),
        "char_count": len(doc.page_content),

    })
    return doc

def hyde_retrieval(query: str, k: int = 3):
    """Generate a hypothetical answer, then use it for retrieval."""
    # Step 1: LLM generates a hypothetical answer
    prompt = f"Write a short passage that answers this question:\n{query}"
    hypothetical = llm.invoke(prompt).content

    # Step 2: Embed the hypothetical answer (closer to doc language)
    docs = vectorstore.similarity_search(hypothetical, k=k)
    return docs

# "How do I cancel?" → hypothetical mentions "termination", "policy", "account"
# → much better match against docs than the raw query

Query Decomposition

Break complex multi-part questions into simpler sub-questions, retrieve for each, then synthesize.

python
def decompose_query(query: str) -> list[str]:
    """Break a complex query into simpler sub-questions."""
    prompt = f"""Break this question into 2-4 simpler sub-questions.
Return as a JSON list of strings.

Question: {query}
Sub-questions:"""
    response = llm.invoke(prompt)
    return json.loads(response.content)

def decomposed_retrieval(query: str):
    sub_queries = decompose_query(query)
    # e.g., "Compare Plan A vs B pricing and features" →
    # ["What is Plan A pricing?", "What are Plan A features?",
    #  "What is Plan B pricing?", "What are Plan B features?"]

    all_docs = []
    for sq in sub_queries:
        all_docs.extend(retriever.invoke(sq))

    # Deduplicate and pass to LLM with original query
    unique_docs = {d.page_content: d for d in all_docs}.values()
    return list(unique_docs)
